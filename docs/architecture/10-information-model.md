# Information Model

## Pipeline

```text
canonical truth
-> registration projection
-> ability format
-> role-specific legal candidates
-> true/false/unreliable constraints
-> Storyteller decision
-> InformationEvaluation
-> InformationDelivered
-> player projection
```

## Orthogonal Information Evaluation

Do not use one `TruthStatus` enum as the only domain fact. Information must be evaluated through separate dimensions:

```text
InformationEvaluation(
  evaluationId,
  abilityEffective,
  semanticTruth,
  reliability,
  truthConstraint,
  registrationApplied,
  registrationDecisionIds,
  simulationReason,
  candidateSetId
)
```

Recommended enums:

```text
semanticTruth:
  TRUE
  FALSE
  NOT_APPLICABLE

reliability:
  RELIABLE
  UNRELIABLE
  NOT_APPLICABLE

truthConstraint:
  NONE
  FORCED_TRUE
  FORCED_FALSE

simulationReason:
  NONE
  DRUNK
  POISONED
  ABILITY_INACTIVE
  PRESENTATION_ONLY
```

This model must express combinations such as:

```text
abilityEffective = false
reliability = UNRELIABLE
truthConstraint = FORCED_FALSE
registrationApplied = true
simulationReason = POISONED
```

That combination covers a case where ability presentation is simulated under poisoning, registration affected the candidate set, and Vortox still forces false Townsfolk information.

## Display Labels

The old Phase One labels may remain as display labels or report summaries:

- `RELIABLE_TRUE`
- `UNRELIABLE_BUT_TRUE`
- `UNRELIABLE_AND_FALSE`
- `ALTERED_BY_REGISTRATION`
- `FORCED_TRUE`
- `FORCED_FALSE`
- `SIMULATED_WITHOUT_ABILITY`
- `PUBLIC_FACT`
- `PLAYER_CLAIM`
- `NOT_APPLICABLE`

They must be derived from `InformationEvaluation`, not stored as the only authoritative fact.

## Candidate Sets

`candidateSetId` points to a hidden candidate set generated by the information engine. Candidate sets must record:

- role-specific output schema;
- true candidates;
- false candidates;
- candidates legal only under unreliable information;
- registration decisions used;
- truth constraints applied;
- Storyteller selected candidate.

Player and AI views receive only the delivered information content, not the full candidate set or truth metadata.

## Drunk And Poisoned

Drunk or poisoned information is unreliable, not automatically false. The engine must still generate legal outputs for that role's format.

Examples:

- Dreamer still receives one good and one evil character.
- Artist still receives a yes/no answer.
- Savant may receive two statements that are not guaranteed to be one true and one false.
- Seamstress may receive yes/no without spending the real ability if drunk.

## Vortox

Effective Vortox forces Townsfolk ability information to be false. This can combine with drunk or poisoned simulation. It requires role-specific false-candidate builders, especially for composite information.

`OQ-V2-004` remains open as an implementation strategy question: composite false-candidate solving may need per-role builders.

## Registration

Registration is evaluated during checks. It may affect apparent role, type, or alignment for information and target legality, but it does not change actual character or actual alignment.

`registrationApplied` and `registrationDecisionIds` must capture whether registration affected the answer and which recorded decisions were used.

## Visibility

Truth metadata is visible to `storytellerView` and `replayTruth`, not to live players or AI. Player and AI views receive only delivered content and legally known context.

## Mathematician Ledger

The Mathematician must use a dedicated `MathematicianAbnormalityLedger`. Do not use a generic counter. The ledger records players whose abilities worked abnormally due to another character's ability, excluding the Mathematician's own abnormal result.

Ledger entries should reference `InformationEvaluation` or mechanical-effect evaluation records instead of inferring abnormality from player-visible labels.
