## abilityRules

1. Mathematician information counts distinct players whose character ability malfunctioned because of another character's ability; it does not count incident records.
2. Multiple qualifying `ABNORMAL` facts for one player contribute exactly one to the count.
3. `NORMAL` contributes zero.
4. `PENDING_TRIGGER` contributes zero and does not block resolution.
5. An `UNRESOLVED` fact blocks a trustworthy number only when it can still alter the distinct-player count.
6. If that same source player already has a qualifying `ABNORMAL` fact, their `UNRESOLVED` fact is redundant and does not block.
7. Only `ABNORMAL` facts with `causedByAnotherCharacterAbility=true` qualify.
8. System information does not qualify.
9. The mere presence of `DRUNK` or `POISONED` does not qualify.
10. An impaired ability whose result is still normal does not qualify.
11. The resolving source player and the resolving current ability instance are excluded from the current count.
12. Exclusion is not a global `roleId=mathematician` filter. A different player's earlier completed Mathematician result may qualify for a later holder if that result is actually `ABNORMAL`.
13. The current delivery cannot feed back into its own count.
14. Historical accepted facts are evaluated from their stored historical classifications and evidence, not reclassified using later current-character state.
15. This delta authorizes only first-night behavior.

## drunkennessRules

- Drunkenness means the source ability may malfunction; it does not mechanically invert a number.
- The mere existence of a drunkenness impairment is not counted.
- If an impaired ability nevertheless produces its normal result, it is not a qualifying abnormal player.
- With no applicable Vortox, the legal Mathematician information candidates for a known drunk source remain the complete fixed domain `0..11`, including the true count.
- The deterministic smallest-false selection is an approved product simulation policy, not an official Storyteller requirement.
- If the true count is `0`, smallest-false selects `1`; otherwise it selects `0`.

## poisoningRules

- Poisoning follows the same ordinary impairment truth boundary: information may be true or false.
- The mere existence of poisoning is not counted.
- A poisoned but normal result is not counted.
- With no applicable Vortox, legal candidates remain all of `0..11`, including the true count; the product policy selects the smallest false value.
- If the current canonical model cannot produce an accepted poisoned source history, tests may cover pure resolver/replay support, but documentation must not fabricate accepted integration reachability.

## VortoxRules

- When an effective Vortox constraint applies, the townsfolk information must be false under the role's information semantics.
- Legal candidates are exactly `0..11` excluding `trueCount`.
- The product simulation policy chooses the smallest false value.
- Vortox plus source drunkenness or poisoning still requires false information; the Vortox false-only constraint remains primary while impairment evidence remains hidden canonical history.
- Vortox identity, role tenure, current character-state revision, role snapshot, and effectiveness must be provable. Ambiguity must fail closed; it must not be guessed.

## unresolvedConflicts

### 2B18B-CONFLICT-001 — required legacy V1 settlement contradicts approved base-first duplicate-holder order

This is a material conflict between accepted canonical history and the approved temporal policy. It cannot be resolved by external BOTC rules because those sources do not define an ordering for multiple Mathematician holders.

Evidence:

1. This slice requires settlement support for legacy Philosopher-gained V1 histories, including coexistence with a base Mathematician.
2. The accepted validator in `packages/domain-core/src/first-night-task-plan.ts:1173-1177` fixes the V1 gained Mathematician task at `baseOrder=100` and `insertionOrder=1`.
3. The accepted validation in `packages/domain-core/src/first-night-task-plan.ts:1307-1318` fixes that task after `PHILOSOPHER_ACTION` and before `MINION_INFO`.
4. The accepted constructor in `packages/domain-core/src/philosopher-ability.ts:825-839` creates the same order.
5. Base `MATHEMATICIAN_INFORMATION` has fixed `baseOrder=1100`.
6. Consequently, any accepted history containing both base Mathematician and a legacy V1 gained Mathematician necessarily makes the V1 gained task canonical-next before the base task.
7. `BOTC-SIM-MATHEMATICIAN-DUPLICATE-HOLDER-TEMPORAL-V1`, and the current Slice 2B18B section 3.5 requirement, instead require base first and gained afterward.
8. V1 is `LEGACY_ACCEPTED_HISTORY`; it may not be automatically migrated, reordered, or rewritten.
9. Skipping the canonical-next V1 task to settle base first violates the canonical next-task contract.
10. Settling V1 in canonical order violates the approved base-first temporal contract.
11. Rejecting this accepted history violates the mandatory legacy V1 settlement support requested for the slice.
12. No official or Chinese rule source resolves this simulator-specific accepted-history contradiction.

Required human decision — choose one explicitly or supply an equivalent precise policy:

- **A — restrict legacy V1:** legacy V1 settlement is supported only for a single holder with no base Mathematician; base+V1 coexistence fails closed.
- **B — approve a V1 duplicate-holder exception:** preserve accepted V1 canonical order for base+V1 histories even though gained resolves before base, with a separately named and versioned temporal policy.
- **C — remove mandatory V1 settlement from 2B18B:** retain V1 replay compatibility but do not require 2B18B to settle legacy V1 Mathematician information.

Until a user-approved choice exists, the rule verdict is `RULE_CONFLICT`. The architect may not design the slice and implementation remains prohibited.
